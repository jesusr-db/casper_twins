# B3 — Absorb caspers-kitchens Event Datagen into Twins DAB

**Status**: Design — pending user review
**Date**: 2026-04-20
**Roadmap reference**: `docs/roadmap-handoff.md` → B3
**Scope**: Phase 0 (deploy script) + Phase 1 (event datagen absorption). Phase 2 data-quality fixes are explicitly out of scope.

---

## Problem

Base Delta tables that twins depends on (`lakeflow.all_events`, `simulator.locations`) are generated by `../caspers-kitchens`. When twins rebuilds its DLT pipeline (e.g. MV → Streaming Table on 2026-04-16), source-table IDs change and Lakebase sync registrations become stale — pointing at deleted instance / pipeline / source-table IDs. The app silently 503's until `create_syncs.py` is re-run manually.

Two symptoms, one root cause: twins does not own the upstream data, and its deploy is not one-command. This spec closes both.

## Goals

1. **Prevent stale-sync failures from recurring silently** by making every deploy self-heal (Phase 0).
2. **Own the event stream** so future pipeline rebuilds don't break source-table references across a bundle boundary (Phase 1).
3. **Keep every artifact deployable via `databricks bundle deploy` or the `setup-lakebase` job** — no orphan scripts, no ad-hoc manual steps.

## Non-goals

- Phase 2 data-quality fixes (duplicate `L1..L6` orders, carryout `delivered` events) — separate future spec.
- Absorbing caspers' LLM-backed data (`complaints.raw_complaints`, `recommender.refund_recommendations`). These stay in caspers because they're `SNAPSHOT` syncs (not fragile to the same failure class) and absorbing them drags in LLM model + serving-endpoint concerns orthogonal to this work.
- CI/CD automation of the new deploy script.
- Multi-environment / multi-workspace support.
- Converting notebooks to `.py` scripts — explicit user decision to import as-is.

## Ownership after this spec lands

| Data | Owner |
|---|---|
| `lakeflow.all_events` | **Twins** (was caspers) |
| `simulator.locations` | **Twins** (was caspers) |
| `lakeflow.orders_enriched`, `driver_positions`, `order_customer_map` | Twins (unchanged) |
| `simulator.customers`, `simulator.customer_address_index` | Twins (unchanged — `generate_customers.py`) |
| `complaints.raw_complaints` | Caspers (unchanged) |
| `recommender.refund_recommendations` | Caspers (unchanged) |
| Menu / supervisor / apps | Caspers (unchanged) |
| Catalog `vdm_classic_rikfy0_catalog` | Shared — both bundles write into it |

After cutover, caspers' complaint/refund agents read events that twins produces. This is an explicit cross-bundle coupling, documented in both repos.

## Architecture & Sequencing

Two independently shippable releases. Each is atomic and reversible.

### Release 1 — `scripts/deploy.sh` (Phase 0)

A single developer-run shell script that chains the three commands already required:

1. `databricks bundle deploy -p DEFAULT`
2. `databricks bundle run setup-lakebase -p DEFAULT`
3. `databricks apps deploy twins-digital-twin -p DEFAULT` (with resolved source path)

Combined with the self-heal already in `setup/create_syncs.py` (2026-04-16), this guarantees stale UC registrations are rebuilt on every deploy. The silent-503 failure class is dead.

### Release 2 — Event datagen absorption (Phase 1)

Import `raw_data.ipynb` and `canonical_data.ipynb` from caspers into twins under `datagen/`. Wire as `notebook_task` entries inside the existing `setup-lakebase` job. Add a new DAB-managed scheduled job `datagen-advance-time` that runs `canonical_data.ipynb` every 3 minutes to advance simulator time. Preserve caspers' env vars (`CATALOG`, `START_DAY=70`, `SPEED_MULTIPLIER=60`, `SCHEDULE_MINUTES=3`).

At cutover: pause caspers' canonical_data job (caspers-side change), unpause twins' `datagen-advance-time`. Two writers must never race.

## Phase 0 — `scripts/deploy.sh`

### New file

`scripts/deploy.sh`:

```bash
#!/usr/bin/env bash
set -euo pipefail
cd "$(dirname "$0")/.."

echo "==> bundle deploy"
databricks bundle deploy -p DEFAULT

echo "==> setup-lakebase (create_syncs self-heals stale UC registrations)"
databricks bundle run setup-lakebase -p DEFAULT

echo "==> apps deploy"
databricks apps deploy twins-digital-twin -p DEFAULT

echo "==> done"
```

Exact form of the `apps deploy` invocation (whether `--source-code-path` is required, or whether the bundle resolves the path automatically) is confirmed in the pre-implementation investigation below.

### Ordering rationale

- `bundle deploy` must land before `setup-lakebase` — the job definition must exist.
- `setup-lakebase` must complete before `apps deploy` — per CLAUDE.md, the Lakebase DB must exist before the app starts (`db.py` crashes otherwise).
- `apps deploy` requires the app in RUNNING state. Script assumes running; developer pre-requisite is documented.

### What does NOT change

- Task order in `databricks.yml` — already correct (`trigger_pipeline` first, then Lakebase tasks). The roadmap's "inversion" bullet is stale.
- `setup/create_syncs.py` — self-heal path already present.

### Constraints

- Hardcoded `-p DEFAULT` per project CLAUDE.md (Azure profile expires frequently).
- No rollback logic — fail fast, let the developer diagnose.
- Developer-run only. No CI.

### Verification

- [ ] `./scripts/deploy.sh` completes on a clean repo.
- [ ] App returns 200 on `/api/health`.
- [ ] `/api/orders` returns rows.
- [ ] Deliberately stale a sync (delete UC registration manually). Re-run `deploy.sh`. `create_syncs.py` self-heal log fires. App recovers.

## Phase 1 — Event datagen absorption

### New directory: `datagen/`

- `datagen/raw_data.ipynb` — verbatim copy of `../caspers-kitchens/stages/raw_data.ipynb` (463 lines). Generates `lakeflow.all_events` and `simulator.locations`.
- `datagen/canonical_data.ipynb` — verbatim copy of `../caspers-kitchens/stages/canonical_data.ipynb` (223 lines). Advances simulator time on each run.

No content changes — preserves caspers behavior exactly.

### Schema management as DAB resources

Declare the four schemas as DAB resources in `databricks.yml` so they're bundle-tracked rather than implicit in notebook code:

```yaml
resources:
  schemas:
    simulator:
      catalog_name: vdm_classic_rikfy0_catalog
      name: simulator
    complaints:
      catalog_name: vdm_classic_rikfy0_catalog
      name: complaints
    recommender:
      catalog_name: vdm_classic_rikfy0_catalog
      name: recommender
    lakeflow:
      catalog_name: vdm_classic_rikfy0_catalog
      name: lakeflow
```

Idempotent — bundle deploy is a no-op if the schemas already exist.

### Consolidated `setup-lakebase` task graph

All provisioning runs through one job. Notebooks are added at the front of the existing graph:

```
raw_data → canonical_data → trigger_pipeline → provision_lakebase
  → generate_customers → create_syncs → finalize
```

Rationale:
- `raw_data` creates `all_events` + `locations` upstream of everything else.
- `canonical_data` ensures `MAX(ts)` is seeded before `trigger_pipeline` starts the DLT pipeline.
- Remaining steps are unchanged from today.

### New scheduled job: `datagen-advance-time`

```yaml
jobs:
  datagen-advance-time:
    name: "twins-datagen-advance-time"
    schedule:
      quartz_cron_expression: "0 */3 * * * ?"
      timezone_id: UTC
      pause_status: PAUSED          # landed paused; cutover flips this
    tasks:
      - task_key: canonical_data
        notebook_task:
          notebook_path: ${workspace.root_path}/datagen/canonical_data
          base_parameters:
            CATALOG: vdm_classic_rikfy0_catalog
            START_DAY: "70"
            SPEED_MULTIPLIER: "60.0"
            SCHEDULE_MINUTES: "3"
```

Env vars mirror caspers exactly. `pause_status` controls ownership — flipped to `UNPAUSED` via a `databricks.yml` edit + `bundle deploy`, never via the UI.

### Cutover sequence (Release 2 deploy day)

Caspers must be paused **before** twins begins writing, otherwise `canonical_data` runs from both bundles compete for the same catalog and may produce duplicate / out-of-order events.

1. `bundle deploy` — lands the notebooks and the job definition with `pause_status: PAUSED`. No writes yet; `datagen-advance-time` is paused, and `setup-lakebase` is not automatically invoked by `bundle deploy`.
2. **Cross-bundle**: caspers team pauses its canonical_data job (caspers-side `bundle deploy` with `pause_status: PAUSED`). Twins writes are still gated by the paused schedule and the un-invoked setup job. Catalog is now quiescent.
3. `bundle run setup-lakebase` — runs `raw_data` (no-op; tables exist) + `canonical_data` (advances once) + remaining provisioning tasks. Twins is now the only writer.
4. Verify `MAX(ts)` on `all_events` has advanced; compare row counts against the pre-pause caspers snapshot.
5. Edit `pause_status: UNPAUSED` on `datagen-advance-time` in twins `databricks.yml`. `bundle deploy`.
6. Watch 30–60 minutes. Confirm `complaints.raw_complaints` and `recommender.refund_recommendations` continue to grow — proves caspers' agents are still consuming the event stream.

**Note on steady-state**: every subsequent `./scripts/deploy.sh` will re-run `setup-lakebase`, which re-runs `canonical_data` as part of its graph — advancing sim-time by one extra tick per deploy. This is harmless (the scheduler advances it again 3 min later regardless) but worth knowing when reading logs.

### Rollback

Reverse steps 5 → 4: flip `pause_status: PAUSED` on twins, unpause caspers. One PR, one `bundle deploy`. Catalog is shared; no data-loss window.

## Destroy-job changes (`setup/destroy_lakebase.py` + `databricks.yml`)

After Phase 1, twins owns `lakeflow.all_events` and `simulator.locations`. Destroy must now remove them too — otherwise a destroy/setup cycle leaks state.

### `setup/destroy_lakebase.py`

- **Pre-step**: use the SDK to set `pause_status: PAUSED` on `twins-datagen-advance-time` — prevents racing with the drop. (Done inline, no new file.)
- **New final step**: drop twins-owned base tables.
  ```python
  BASE_TABLES = [
      f"{SOURCE_CATALOG}.lakeflow.all_events",
      f"{SOURCE_CATALOG}.simulator.locations",
  ]
  ```
  `DROP TABLE IF EXISTS` each. Runs AFTER synced-table drops and Lakebase destruction (synced tables consume the base tables).

### Updated destroy order (final)

1. Pause `datagen-advance-time` via SDK.
2. Drop synced tables (existing).
3. Destroy Lakebase instance (existing).
4. Drop DLT-produced tables (existing).
5. Drop datagen-produced base tables (new).

### Header comment update

Remove the stale line about caspers owning source Delta tables; replace with explicit ownership of `all_events` + `locations`. Keep explicit "does NOT touch" for `complaints.raw_complaints` and `recommender.refund_recommendations` — those stay caspers-owned.

### Verification

- [ ] `bundle run destroy-lakebase` → `bundle run setup-lakebase` cycle works end-to-end.
- [ ] App comes up with fresh event stream; `MAX(ts)` starts from the `START_DAY=70` seed.

## DAB / setup-job deployability audit

Every artifact reaches production through (a) `databricks bundle deploy` declaring DAB resources, or (b) a task in the `setup-lakebase` job. No ad-hoc scripts, no direct SQL from a laptop, no manual UI pokes.

| Deliverable | Path |
|---|---|
| `scripts/deploy.sh` | Orchestrator only — wraps three CLI commands that operate on DAB/job resources. No SQL, no direct API calls. |
| `datagen/raw_data.ipynb`, `canonical_data.ipynb` | Uploaded via `bundle deploy`. Executed as `notebook_task` entries inside `setup-lakebase`. |
| Schemas (`simulator`, `complaints`, `recommender`, `lakeflow`) | Declared as DAB `schemas` resources in `databricks.yml`. |
| `datagen-advance-time` scheduled job | DAB-declared. Cutover via `pause_status` edit + `bundle deploy`. |
| Notebook env vars | `base_parameters` on the job task in `databricks.yml`. |
| Destroy symmetry | `setup/destroy_lakebase.py`, invoked by the DAB-managed `destroy-lakebase` job. |
| Cutover (caspers side) | Out of twins' scope — caspers owns its own `pause_status` edit + deploy. Tracked as a coordination item. |

**Operational surface after Phase 1**:
- One setup job: `twins-setup-lakebase` (idempotent; runs on every deploy).
- One destroy job: `twins-destroy-lakebase` (idempotent; symmetric to setup).
- One scheduled job: `twins-datagen-advance-time` (3-min cadence; paused/unpaused via DAB).
- One app: `twins-digital-twin`.
- One shell script: `scripts/deploy.sh` (developer entrypoint; three CLI calls).

## Pre-implementation investigation (land as first tasks in the plan)

1. **Read `raw_data.ipynb` in full** — confirm it's idempotent on existing tables (`CREATE TABLE IF NOT EXISTS` or equivalent). Enumerate every env var it reads and every schema/table it creates.
2. **Read `canonical_data.ipynb` in full** — confirm time-advancement logic and any external dependencies (UC volumes, secrets, endpoints).
3. **Confirm `apps deploy` command shape** — whether `--source-code-path` is required with bundle-deployed apps or whether `databricks apps deploy twins-digital-twin -p DEFAULT` alone is sufficient after `bundle deploy`.
4. **Confirm `bundle run` vs `jobs run-now`** — prefer `databricks bundle run setup-lakebase` as the idiomatic DAB invocation.
5. **Confirm DAB `schemas` resource shape** — validate the YAML block against current DAB schema reference; may need `comment` or `owner` fields.
6. **Pre-flight `SHOW SCHEMAS IN vdm_classic_rikfy0_catalog`** — diff against what `raw_data.ipynb` creates, to catch any schema twins doesn't yet know about.

## Risks

| Risk | Likelihood | Mitigation |
|---|---|---|
| Two schedulers race during cutover | Medium | Strict step ordering; caspers pause must land **before** twins unpause. Runbook enforces. |
| `raw_data.ipynb` not idempotent on re-run | Medium | Pre-implementation investigation #1. If non-idempotent, wrap with `CREATE TABLE IF NOT EXISTS` guards at the edges (minimal notebook edit). |
| Env var drift caspers → twins | Low | Mirror all vars explicitly in `base_parameters`. Test in a clean workspace. |
| Schema DDL conflict (e.g. caspers created schema with different owner) | Low | DAB `schemas` resource is idempotent — will no-op on existing schemas. Pre-flight diff in investigation #6. |
| `apps deploy` command shape wrong | Low | Investigation #3. Script fails fast if shape is incorrect. |
| Notebook task `${workspace.root_path}` resolution | Low | Confirm DAB uploads `.ipynb` to the bundle root with an importable path. |
| Cross-bundle coordination failure | Medium | Explicit runbook; caspers pause is a documented dependency, not a silent assumption. |
