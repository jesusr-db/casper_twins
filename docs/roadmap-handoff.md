## Backlog

### B1 — Missing driver pin during pickup-to-first-ping gap

**Priority**: Low
**Effort**: Medium (backend query change)

**What happens**: When a driver picks up an order, there is a ~60-second window before the first `driver_ping` event fires. During this window:
- The order correctly shows as "In Transit" in the pipeline bar and order drawer
- `Drivers Out` KPI increments correctly
- **But no red driver dot appears on the map** — because driver markers are sourced from `driver_positions_synced`, which is only populated by `driver_ping` events, not `driver_picked_up`

**Root cause**: `driver_positions_synced` is a streaming table fed by `driver_ping` events. There is no entry until the simulator emits the first ping (~60s after pickup). The `driver_picked_up` event alone does not create a position entry.

**Proposed fix**: Add a `UNION ALL` to the drivers backend query that returns orders where `picked_up_at IS NOT NULL AND delivered_at IS NULL AND NOT EXISTS (SELECT 1 FROM driver_positions_synced WHERE order_id = oe.order_id)`, positioned at the store's lat/lon as a fallback. Show these with `progress_pct = 0`.

**Why not done yet**: First attempt at the UNION caused a 500 — the `NOT EXISTS` subquery referencing `driver_positions_synced` without a location/time filter caused a full table scan + type error. Needs a scoped subquery to be safe. Reverted to keep the app stable.

**Workaround**: None needed — the gap is ~60 seconds and resolves itself when the first ping arrives.

### B3 — Absorb caspers-kitchens **event datagen** into twins DAB

**Priority**: Medium
**Effort**: Large (multi-day, phased)

**What's wrong today**: Base Delta tables twins depends on (`lakeflow.all_events`, `simulator.locations`) are generated by `../caspers-kitchens`. When twins rebuilds its DLT pipeline (e.g. MV → Streaming Table, 2026-04-16), the Delta source IDs change and Lakebase sync registrations become stale (pointing to a deleted instance/pipeline/table). App silently 503's until `create_syncs.py` is re-run manually.

**Concrete failure 2026-04-16**: `orders_enriched_synced` UC registration referenced a deleted Lakebase instance (`94f17a56-87c5-4e5f-b27a-778849a37d88`), deleted sync pipeline (`2fb431fd-83df-4500-88bb-96de146e847b`), and old source-table ID. All data-bearing endpoints returned 503 `UndefinedTable`. Fixed by deleting the stale UC registration and re-running setup-lakebase; `setup/create_syncs.py` also hardened to self-heal on "not found" errors.

**Scope (narrowed from original B3)**:

| | Absorb into twins | Stay in caspers |
|---|:---:|:---:|
| `lakeflow.all_events` generator (`raw_data.ipynb`, `canonical_data.ipynb`) | ✅ | |
| `simulator.locations` (88 stores) + schema DDL | ✅ | |
| Continuous sim-time advancement mechanism | ✅ | |
| `complaints.raw_complaints` generator + LLM agent | | ✅ |
| `recommender.refund_recommendations` generator + LLM agent | | ✅ |
| Menu Genie, supervisor, apps | | ✅ |

**Why leave complaints + refunds in caspers**:
- LLM-generated via serving endpoints (`${catalog}_complaint_agent`, `${catalog}_refund_agent`) — absorbing drags in LLM model config, endpoint deployment, and serving-side secrets orthogonal to twins.
- Both sync as `SNAPSHOT`, not `CONTINUOUS` — rebuilt cleanly on every `create_syncs` run, so they are **not fragile** to source-table-id churn the way `orders_enriched_synced` was.
- Today's outage was on a twins-owned sync; fixing that class of bug doesn't require owning LLM-generated data.

**Phase 0 — `scripts/deploy.sh` (do now, ~1 day)**
```bash
#!/usr/bin/env bash
set -euo pipefail
databricks bundle deploy -p DEFAULT
databricks jobs run-now twins-setup-lakebase -p DEFAULT --wait
databricks apps deploy twins-digital-twin -p DEFAULT
```
Combined with the self-heal logic already in `create_syncs.py` (added 2026-04-16), this guarantees stale syncs get rebuilt on every deploy. **Prevents today's failure from recurring.** Mandatory pre-req for Phase 1.

**Phase 1 — absorb event datagen (multi-day)**
- Import `stages/raw_data.ipynb` + `stages/canonical_data.ipynb` into twins (new `datagen/` directory or folded into `pipelines/`).
- Import the mechanism that continuously advances `MAX(ts)` in `all_events` — saw it move mid-session today (`04:10:35` → `04:39:42` over ~30 min); need to identify the caspers job/schedule responsible and port it.
- Add `simulator.locations` + schema DDL (`complaints`, `recommender`, `simulator` schemas) to twins setup. Currently implicit in caspers `raw_data.ipynb` + `init.ipynb`; `vdm_classic_rikfy0_catalog` is assumed to exist.
- **Critical task-ordering inversion** in `databricks.yml`:
  - Current: `provision → generate_customers → create_syncs → finalize → trigger_pipeline`
  - After:   `provision → trigger_pipeline → generate_customers → create_syncs → finalize`
  - Reason: `generate_customers.py` clusters real delivery addresses from `orders_enriched_synced`, which depends on the DLT pipeline having run.

**Phase 2 — optional data-quality fixes while in datagen code**
- Eliminate loop-duplicate orders (`-L1..L6` variants replayed every ~3 months with identical content — see today's fix in `backend/routes/orders.py` that papers over these with a 24h recency filter).
- Emit proper `delivered` events for carryout orders — currently `driver_picked_up` body is `{}` and no subsequent events fire. Twins has a body-shape detection workaround in `pipelines/orders_enriched.sql`; cleaner to fix at the simulator.

**Cross-bundle dependencies retained (explicit)**:
- Twins reads `vdm_classic_rikfy0_catalog.complaints.raw_complaints` — caspers must keep deploying this + the `${catalog}_complaint_agent` serving endpoint.
- Twins reads `...recommender.refund_recommendations` — caspers must keep deploying this + the `${catalog}_refund_agent` serving endpoint.
- **Data-quality note**: `c.ts` in complaints is clamped to Jan–Mar 2026 regardless of simulator time. Not fixed by this migration; requires caspers-side change to complaint generator. Twins worked around by filtering complaints by `oe.created_at` instead of `c.ts`.

**Twins-native data not affected by this migration**:

- DLT outputs `orders_enriched`, `driver_positions`, `order_customer_map` (already twins-owned)

### B2 — Timezone: SLA timers show orders as 8 hours old ✅ Fixed 2026-03-19

**Priority**: High (UX-breaking)
**Effort**: Low (1-line frontend fix)

**What happens**: Orders display as "480 min in New" — SLA timers count from 8 hours ago instead of the actual order age. Affects all `getMinutesInStage` calculations and SLA dot colors.

**Root cause**: `parseTimestamp` in `sla.ts` only appended "Z" (UTC) for space-separated timestamp strings. Timestamps returned by Lakebase via asyncpg are serialized by FastAPI as `"YYYY-MM-DDTHH:MM:SS"` (with "T", no "Z"). `new Date("2026-03-19T10:00:00")` without "Z" is parsed by browsers as **local time**, not UTC. Users 8 hours ahead of UTC see all timestamps interpreted 8 hours earlier than actual.

**Fix applied**: `parseTimestamp` now normalizes both formats to ISO + always appends "Z" unless the string already carries timezone info (`Z` or `±HH:MM` suffix). File: `frontend/src/constants/sla.ts`.
